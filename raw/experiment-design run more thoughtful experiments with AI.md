I think many inconclusive experiment results start as design failures. You never thought about the result that would stop the work, so almost any outcome can be made to support the old belief.

I know this is basic af, but happens all the time everywhere.

## The fix is easy, or is it?

The fix is a one-page experiment design sheet. Filling it in before launch, while the team can still change the plan.

Start with the hypothesis. Write down the result that would prove it wrong.

Then pick the cheapest method that could answer the question. Name one primary metric and the smallest effect worth acting on.

Check whether the sample size or run time is feasible. Then write the decision rules: scale, stop, revise, inconclusive, and unacceptable harm.

## The hard part is doing it

Experiment design is full of judgment calls. You have to decide which result would matter, which effect would be worth chasing, and which outcome should end the idea. A template cannot make those calls.

The problem is that the product manager is almost always the person driving this, and they might even have the answers in their head.

But there isn’t enough time to think it through properly or put it in the right order.

Now imagine that’s just one experiment. What if it’s ten?

If everything is an experiment in the agentic age, this becomes a daily thing you have to run. And of course, you can’t do that in your head in the moment - or run all the mental calculations. Even a template wouldn’t be easy to fill out.
