# Kivora

> An AI-powered multi-tenant SaaS GRC platform — risk register, compliance frameworks, governance lifecycle, and vendor-trust visibility, with a Claude-driven agent that queries workspace data via tools and answers via SSE streaming.

## What and why

Kivora (product name **ROAM GRC**) addresses the lack of a single, AI-aware platform for governance, risk, and compliance teams who currently juggle spreadsheets, isolated GRC tools, and ad-hoc vendor questionnaires. The platform unifies six core domains — risk management, compliance (frameworks + controls + evidence), governance (policies/attestations/exceptions), vendor trust, task management, and AI services — under a multi-tenant architecture with per-workspace scoping. Subscription tiers (`STARTUP`, `GROWTH`, `ENTERPRISE`, `SYSTEM`) gate feature access. The Python FastAPI agent reasons over real workspace data using **20 tool definitions** (14 base + 6 Composio-gated for remediation and PII discovery), persists conversations with Haiku-driven compaction at 50+ messages, and enforces RBAC via a per-tool capability matrix that filters tools at registration time so Claude only sees what the caller's role permits. The hybrid Java + Python AI architecture reserves complex reasoning/chat for the Python agent and routes simple AI tasks through a Spring AI proxy.

## Architecture decisions

- **2026-02** — **Modular monolith over microservices** for backend. 13 Maven modules, single Spring Boot deploy. Same pattern as [[wiki/projects/vedge|Vedge]]. Rationale: small team, transactional consistency, deploy simplicity.
- **2026-02** — **Two-layer tenant isolation**. Layer 1: Hibernate `@Filter` on `TenantScopedEntity` base class. Layer 2: PostgreSQL Row-Level Security policies. `TenantContext` is a `ThreadLocal` set from JWT claims by `TenantSecurityFilter`. Workspace selected per-session via `X-Workspace-ID` header. Defense in depth — RLS catches anything that bypasses Hibernate.
- **2026-02** — **Hybrid Python + Java AI architecture**. Python FastAPI agent (`agent/`) handles complex reasoning, tool use, and RAG with Claude Sonnet. Java `kivora-ai` module handles simple AI tasks (risk-mitigation suggestions, policy drafts) via OpenAI and acts as an authenticated proxy. Mirrors [[wiki/concepts/reasoning-execution-split]].
- **2026-02** — **In-process SQL agent tools, no HTTP round-trips**. The Python agent's 12 tools execute SQL directly against GRC tables (tenant-scoped) rather than calling the Java backend's REST API. Tradeoff: tighter coupling between Python and DB schema, but lower latency and no auth replay between services.
- **2026-02** — **PostgreSQL + pgvector instead of a dedicated vector DB**. RAG uses 384-dim MiniLM-L6-v2 embeddings stored in `ai_document_chunks` with HNSW index. Chunking is sentence-boundary, 500 chars, 50-char overlap. Top-5 cosine search.
- **2026-02** — **SSE streaming over WebSockets** for chat. `sse-starlette` async generators yield `text` / `tool_use` / `tool_result` / `done` events.
- **2026-02** — **Conversation compaction at 50+ messages** by Haiku, replacing older messages with a summary. Bounds context cost.
- **2026-02** — **Subscription-tier feature gating** via `requireFeature()` in services/controllers + `user.features` filter on the frontend. Currently incomplete (see Open questions).

## Lessons learned

Distilled from `tasks/lessons.md` and Kivora's auto-memory. Promote to `wiki/concepts/` if any generalize beyond GRC.

**Frontend / UX (L001-L008):**

- Navigation item count > completeness — 7 items beats 15; use section landing pages + tabs.
- Dual entry points for the same feature break user mental models — pick one primary surface and commit.
- Dead buttons destroy trust faster than missing features — remove unwired UI immediately.
- "Dashboard" proliferation signals unclear ownership — domain dashboards belong as section landing pages, not peer nav items.

**Engineering (E001-E005):**

- `import.meta.env.VITE_*` is baked at build time — adding env vars in Vercel post-deploy doesn't fix a deployed bundle without a rebuild.
- Vercel doesn't auto-rebuild on env var changes — explicit redeploy required.
- Stale Maven jars cause Spring 404s — when adding a controller in a sub-module, run `mvn install -pl <module> -am -DskipTests` before booting from `kivora-app`.
- Hardcoded fallback URLs (`|| 'http://localhost:8080'`) are a production footgun — use `|| ''` so failures surface visibly.

**Workflow (from auto-memory):**

- **Always run review/QA agents before pushing.** The pattern of push → user finds crash → fix → push has been wasteful. A code-reviewer scan catches snake_case/camelCase mismatches, null-guard gaps, API path mismatches, missing array defaults, wrong HTTP methods, JdbcTemplate column errors, and LazyInitializationException patterns before they ship.

**Engineering — Findings migration (2026-05-08):**

- **`ALTER COLUMN ... USING ::text` doesn't rewrite expressions in dependent objects.** When converting a PostgreSQL native enum to VARCHAR, CHECK constraints and partial-index `WHERE` clauses that referenced the enum type must be dropped first and recreated against the text-typed column. Surfaced in `V097__findings_enums_to_varchar.sql` after `V096` shipped native enums and Hibernate enum mapping turned out to be awkward.
- **Dual-write rollout demands swallowed-failure semantics in the new path.** A bug in a parallel write must never cascade into an existing customer-facing flow. Pattern: `try { ingest() } catch (Exception e) { log.warn(...); }` — loud enough for operators to investigate, silent to customers. Generalized at [[wiki/concepts/dual-write-rollout]].
- **The "inverse contract" — explicit `invalid-*` fixtures that MUST fail — is what proves a conformance test bites.** A test that only asserts "valid things validate" silently degrades to a no-op as soon as the validator breaks. Verified during Phase 6 by deliberately making the invalid fixture valid and watching the test fail.
- **A synthetic-connector smoke harness pays for itself within one phase.** Adding `SyntheticFindingsConnector` (Phase 4) before migrating Composio (Phase 5) cost one week and converted Phase 5 from "discover bugs in the pipeline" into "a pure mapping exercise." It also stays as a permanent smoke test for debugging future ingestion regressions.

**Engineering — Systems Inventory (2026-05-16/17):**

- **PG JDBC binds untyped null String parameters as `bytea`.** The expression `LOWER(s.name) LIKE LOWER(CONCAT('%', :search, '%'))` fails at PREPARE time with `function lower(bytea) does not exist` when `:search` is null — because `CONCAT(text, null::bytea, text)` evaluates to `bytea`, and `LOWER(bytea)` is not a defined Postgres function. The `(:search IS NULL OR ...)` short-circuit pattern doesn't help because Postgres type-checks the whole expression at PREPARE time, before any short-circuit evaluation. **Fix:** drop the `IS NULL` check entirely; service layer normalizes null/blank search to `""`; empty string matches every row via `LIKE '%%'`. Same defensive pattern applied preemptively to TEXT[] params in Wave 4 via `CAST(:environments AS text[])`. Surfaced live on staging Wave 2; documented in `SystemRepository.findFiltered` javadoc.
- **Frontend permission gates must match what the backend actually enforces, not what feels consistent.** Wave 2 followed the `RiskListPage` convention of `hasPermission('system:write')` for the New System CTA. But `risk:write` is registered in the platform permissions matrix; `system:write` is not — the Systems backend gates on roles via `@PreAuthorize`, not permission strings. `user.permissions.includes('system:write')` returned `false` for every user including `PLATFORM_ADMIN`, hiding the entire Create flow. **Fix:** gate UI on roles (`hasRole('WORKSPACE_ADMIN')`) when the backend gates on roles. The role-vs-permission inconsistency across Kivora modules is a real footgun for any new feature; the inconsistency itself is worth fixing platform-wide in a future cleanup.
- **Cross-module dependency avoidance via native SQL on raw tables is cheap and pays off.** Wave 3 needed to display Risk fields on the Systems detail page; Wave 4 needed Framework name; Wave 5 needed Vendor display fields. Adding a Maven dep from `kivora-compliance` onto each adjacent module would have created a deep coupling web — and `kivora-compliance` already owns the junction tables, so the natural place for these queries is the Systems module repos. Native SQL queries (`SELECT ... FROM risks JOIN ...` etc.) keep the module graph clean, surfaced table-name realities (`risks` not `risk_register`; `master_frameworks` not `frameworks`; `framework_adoptions` not `adopted_frameworks`) that the PRD's idealized names would otherwise have missed. Tradeoff: schema renames in adjacent modules silently break these joins until a real run hits them — worth a periodic grep, but no worse than the JPA-only alternative would have been.

**Engineering — Tier 2A end-to-end cutover (2026-05-17/19, Wave 7–41):**

- **When two enums genuinely encode different facts about the same row, the answer is additive (two columns), not unification.** ADR-0002 Path 3 is the worked example. `ControlInstanceStatus` (workflow: `NOT_STARTED → IMPLEMENTED → FAILED`) and `ProjectedControlStatus` (assessment: `PASS / FAIL / INCONCLUSIVE / NO_RECENT_FINDINGS`) had three downstream consumers with three legitimately different translation tables (Wave 19 DiffLogger, Wave 20 dashboard Assembler, Wave 21+ planned writer) — the "three rules for divergence" finding. Unifying into one 8-value enum forces one of those three intents to silently accept the wrong tradeoff; translating forever at every read site multiplies the truth tables. Materializing both columns with a single alignment-rule function (`lifecycleAndAssessmentAligned`) makes the distinction structural and indexable. Generalizes to any time a data model is asked to encode workflow AND evidence — they're different facts.
- **Verify-claims cadence catches its own undercount.** Wave 31 cutover claimed read-path completion; Wave 32's flag-audit cadence used `grep` + `git diff` against the prior wave's claim file and found a 7th site that Wave 31 had missed (`DashboardInsightsService`). Wave 33 closed it. Wave 34 then added a Flyway-replay harness + pinning tests that programmatically catch the pattern (PG state after running V1..VN matches a golden file). The discipline: every claim of "all sites updated" is verifiable, and the audit pass that verifies it is itself a tracked artifact.
- **Flag-audit cadence as recurring discipline.** Every 5–6 waves run a `chore(audit): flag-audit cadence` cycle that reviews every live feature flag (per-tenant overrides, kill switches, rollout gates). Wave 22 caught the `TenantPlanResolverAdapter` cache-eviction P0 (a feature catalog mutation didn't bust the cache). Wave 27 surfaced a P1 architecture gap — Tier 2A had only a global env-var flag, no per-tenant override mechanism — which drove V106. Wave 32 caught Wave 31's read-path undercount. Wave 37 ran clean and confirmed the 3 process improvements paid off. The pattern: cadence-not-event audits force latent issues to surface within bounded time, instead of after a customer incident.
- **Lift error-swallowing in fetchers before bridging to a multi-status contract.** The Tier-1 Finding schema's whole point is to surface `INCONCLUSIVE` ("we couldn't check") as distinct from `FAIL` ("we checked, it failed"). But a `try { ... } catch (Exception e) { return List.of(); }` in a fetcher silently degrades `INCONCLUSIVE` to "we got nothing" — the bridge never sees the failure. Commit `9097003` lifted error-swallowing in 9 connector fetchers in one pass. **Generalization:** if downstream code distinguishes "no signal" from "no signal because we failed," fetchers must propagate the distinction. The contract type carries the meaning, but the fetcher has to provide the input.
- **Port-based abstraction is what makes a cross-module read-path cutover one-PR-shaped.** `ControlStatusSourcePort` + `ControlStatusSourceResolver` (Wave 33) replaced what would otherwise have been a 5-module sprawl. Each consumer (DashboardInsightsService, ComplianceFrameworkController, GapAnalysisService, AnomalyDetection, RiskDiscoveryScheduler) reads from the port; the port decides lifecycle-vs-assessed based on the tenant's `Tier2AMode`. The infra cost (defining the port and the resolver) bought the ability to add per-row projection sweeps (Wave 38) cleanly. **Generalization:** when a value's *source* is changing across the codebase, route every read through a port — don't propagate the conditional to every call site.

**Engineering — Cross-environment publishing infrastructure (2026-06-07/08):**

- **Spring's relaxed property→env binding silently consumes the wrong env var.** `@Value("${kivora.promote.signing-key-hex:}")` binds to env `KIVORA_PROMOTE_SIGNING_KEY_HEX` (dots+dashes → underscores, all uppercased). The W2 javadoc documented the env var name as `PROMOTE_SIGNING_PRIVATE_KEY_HEX` — close enough to look right at a glance, completely wrong from Spring's perspective. Operator set the env vars per the (incorrect) docs; `PromoteSigner` logged DISABLED at boot; cross-env promote requests sat in REQUESTED forever because approve would fail `ERR_SIGNER_DISABLED`. **Generalization:** when documenting an env var name in code, derive it from Spring's binding rules, not from intuition. Better: add a startup self-check that logs the bound property names alongside what env vars are visible, so operators see at boot whether their values landed in the right place.
- **Merge resolution can SILENTLY DROP a line both branches consented to remove.** Today's chat.py 3-way merge had explicit conflict markers in 3 hunks (docstring, logger format, agent_core.run kwargs). I resolved kept the rich auth-aware version with `principal=...` in the call. But the line that EXTRACTS `principal = getattr(request.state, "principal", None)` was missing from BOTH branches — neither had it explicitly, both implicitly assumed the other did. The merge resolved with conflict markers gone but the `principal` variable undefined. Every chat request in prod hit `NameError: name 'principal' is not defined` for ~14h before someone tried to chat. **Generalization:** when a merge resolution keeps a USE of a variable, grep for the declaration in the resolved file — don't assume the surrounding context still defines it. Better: a smoke test that hits `/chat/stream` post-deploy and asserts the first SSE chunk is not a `{"type":"error"}` event.
- **The "empty manifest" guard correctly refuses M2M-mapped frameworks if the builder query is M2O-only.** Same architectural class of bug as the slowapi param-order issue — a downstream check is correct but the upstream query is wrong, so the check fires on legitimate input. The fix is always upstream (extend the query / fix the param order), never relaxing the downstream guard. Generalization: when a defensive guard rejects what looks like valid input, the bug is almost always in the data the guard sees, not in the guard's logic.
- **Backend list endpoints with multi-admin users must filter per-caller.** `MasterFrameworkPromotionService.listRequests(status)` returned ALL requests of the given status — not just ones where the caller was requester or approver. The W5 AdminShell badge code had a comment claiming "Backend already scopes to current admin" — that comment was untrue from day one. Surfaced when Richmond went looking for a request Godwin had assigned to him and saw an empty inbox (because the badge was counting Godwin's request, but the list page hadn't filtered correctly). Fix: `@Query`-annotated `findVisibleTo(userId)` + `findVisibleToWithStatus(userId, status)` with explicit JPQL `WHERE r.requestedBy = :userId OR r.approverUserId = :userId`. **Generalization:** never trust a code comment that says "backend filters X" — verify by reading the query.
- **`INNER JOIN` to a parent table excludes orphans of an intentionally-nullable FK.** V083 made `master_questionnaire_questions.template_id` nullable so questions can exist standalone (a question bank that pre-dates assembling them into templates). The PlatformContentManifestBuilder still INNER-JOINed templates → 95 standalone questions silently excluded from the snapshot. V138 added a stable `code` column on questions + switched builder to LEFT JOIN + UPSERT-by-code. **Generalization:** if a column was made nullable as a feature evolution, every query that joins to its parent must be reviewed for INNER-JOIN-induced exclusion.
- **Multi-repo PR-style staging→main promotes need `git push origin staging:main` only after absorbing prior PR merge commits.** Every prior staging→main was via GitHub PR, which creates a merge commit on main that doesn't exist on staging. Pushing staging directly to main is rejected as non-fast-forward. Workaround: `git merge origin/main` into local staging first (absorbs the PR merge commits), then `git push origin staging:main` succeeds as fast-forward. Backend + frontend merged clean today; agent had a real chat.py conflict that needed careful resolution (lesson above). **Generalization:** if the team's main-promote workflow is PR-based, the same workflow has to be honored on direct pushes — either go through a PR or absorb the merge commits locally first.

**Operational — DB role cutover incident (2026-06-07 22:05–22:15, 10-min prod outage):**

- **Setting Railway `DB_USER` to a new value without first creating that role in Postgres = guaranteed crash-loop.** Postgres roles are application-managed; Railway lets you set any username + password as env vars regardless of whether they resolve in the actual DB. The connection attempt fails at startup → Flyway init fails → Spring Boot exits → Railway healthcheck fails → restart loop. Both live-api and live-agent crashed within seconds of the env-var change. **Generalization:** for a DB role/permission change, the order is always (1) make the change in the DB itself, (2) verify a connection works with the new credentials, (3) then update the env vars on the application. The other order is a crash.
- **A "password rotation" that only changes env-var labels is not a rotation.** Confirmed when the recovery path of setting `DB_USER=postgres` with the "new rotated" password failed — Postgres still accepted only the original (leaked) password. The actual `ALTER USER postgres WITH PASSWORD` had never been issued; the original leaked credential was the only one Postgres would honor. **Generalization:** rotation = ALTER USER in the DB + synchronous env var update on every consumer. Anything less is rotation-shaped theater that fails on first failover.

**Engineering — Compass expansion + QA sweep (2026-06-24/25):**

- **Stale QA reports drive redundant workflow phases.** The 2026-07-13 GRC-100% workflow allocated an item for "AttestationController + 6 un-stubs" based on a prior QA report. The plan-agent grepped for `FEATURE_NOT_ENABLED` and found only one hit (`_request_attestation`, a different tool), plus verified `attestations` + `auditor_attestations` tables already live in V008/V091 with a working controller + service. Pre-flight plan agent with cheap grep/read authority killed an entire impl-phase's worth of redundant work. **Generalization:** when a workflow's plan phase returns `in_scope=false` with concrete file evidence (paths + line numbers), trust it — don't re-litigate scoping in the impl phase. Cheap plan agents are the natural place to catch stale premises.
- **NotificationType strings are silently swallowed by the Java enum on the receiving side.** The 2026-06-25 QA sweep found 11 typos where agent tools dispatched notifications with type strings not in the Java `NotificationType` enum (`POLICY_ATTESTATION` when the enum only had `ATTESTATION_DECLINED` and `GENERAL`; `RISK_UPDATED` when the enum had `RISK_STATUS_CHANGED`; etc.). The `/admin/api/v1/internal/notifications` endpoint accepted the request but the dispatcher couldn't route it — every fan-out was a silent no-op. **Generalization:** when a string parameter has a closed enumeration on the receiving side, either validate at the API boundary (400 on unknown values, don't guess) or accept that misspellings are silent no-ops. A `List.of(NotificationType.values())` enum-check at the controller edge is the cheapest form of the fix.

**Engineering — account migration + Railway recovery (2026-07-23):**

- **Docker-test locally before every deploy push.** Seven failure layers surfaced one ~10-minute Railway cycle at a time until the local rig (docker build + boot against fresh `pgvector/pgvector:pg16` + `redis:7-alpine`) replayed the entire chain in one pass: full Flyway migration V001→V173 AND Hibernate schema validation. Compounding trap: the Railway CLI truncated build logs to their first line, which misdiagnosed straightforward pip conflicts as builder-infrastructure failures (even Railway's own diagnostic agent agreed with the wrong theory). **Generalization:** environment parity locally is cheap; iterating through a remote build queue is not — and never trust a truncated log to name the failure.
- **Hibernate 6.5 resolves `List<String>` basic attributes to the JSON JdbcType even under `@JdbcTypeCode(SqlTypes.ARRAY)`.** Schema validation then fails against a real `text[]` column. Fix: `String[]` entity fields (natively validated as ARRAY) with hand-written List-preserving accessors so call sites don't change. Corollary: staging validates schema while prod doesn't — staging is the canary for entity/DDL drift, and a prod-boots-but-staging-fails split means the validator caught something real, not that staging is broken.
- **Migration seeds must tolerate per-environment catalog state.** V170's hard `RAISE EXCEPTION` on unknown framework slugs crashed prod because prod's master catalog arrived via the promotion pipeline with different slugs than the V018 dev seed. Prune-with-`RAISE WARNING` is the pattern: each environment keeps only references its own catalog resolves. Same family as the M2M manifest lesson — assumptions about environment-invariant data are migration bugs waiting to fire.
- **`CREATE INDEX CONCURRENTLY` migrations self-deadlock on stock Postgres.** V118 hung against the local test DB: CONCURRENTLY waits for all older snapshots, including Flyway's own second connection idle-in-transaction. Managed Postgres masks this with `idle_in_transaction_session_timeout`; stock images never time out. Set it (30s) on any local/test rig replaying migrations.
